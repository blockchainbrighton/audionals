Overview

The TEAM methodology enables an LLM-based AI assistant to build large-scale JavaScript blockchain web apps despite a strict 1,000-line context limit.

The Problem

Building complex blockchain applications with AI assistance faces a fundamental limitation: AI models have context windows typically limited to ~1,000 lines of code. This makes it challenging for AI to maintain coherence across large codebases.

The Solution

TEAM decomposes applications into behavioral modules or services (e.g., UserAuth, PaymentProcessing, BlockchainInscriber, UI components), each with a well-defined interface. This modular approach allows AI to focus on one component at a time while maintaining system-wide compatibility.

Key Principles

Interface Standardization & Versioning

Each module's API (function signatures, data schemas, events) is specified in advance and changes only via explicit version bumps. This mirrors microservices best practices: well-defined APIs let one module be replaced or upgraded without breaking others.

Module Isolation

Each module lives in its own folder with clear boundaries. We follow AOMA principles: code and comments are structured for "easy parsing by language models", and interfaces are immutable by default. This ensures AI can focus on one module at a time without losing coherence.

Human Oversight

Non-technical supervisors play the roles of architect and project manager. They specify and review but don't write code. The TEAM tool translates business requirements into AI prompts, and presents summarized outputs for human review and approval.

Transparent Documentation

Throughout the TEAM process, full documentation and tests are produced alongside code. The LLM is prompted to include detailed comments (WHY/INPUT/OUTPUT) as in AOMA's "documentation-as-code". This ensures maintainability and transparency.

Module Definition

A human overseer uses a simple schema (e.g., YAML/JSON) to declare a new module's purpose, inputs, outputs, and dependencies on other modules. For example: "Module UserAuth: inputs (username, password), output (JWT token); depends on DatabaseService."

# Example Module Definition
name: WalletConnector
description: Connects to blockchain wallets
inputs:
  - walletType: string
  - network: string
outputs:
  - connection: WalletConnection
  - address: string
dependencies:
  - Web3Service

AI Prompting

The orchestrator builds a prompt containing the module's specification and relevant interface definitions. It restricts the input to ‚â§1000 lines by including only the new module's context, not the entire codebase. The LLM then produces the code, documentation, and tests.

// AI Prompt Example
"Implement a JavaScript class WalletConnector given this API spec:
- Method: connectWallet(walletType: string, network: string)
- Method: signTransaction(tx: Transaction)
- Uses Web3Service.getProvider()
Include JSDoc comments and Jest tests."

Automated Checks

The tool inserts the code into the project and runs the CI pipeline: it type-checks or lints, and runs the generated tests. If tests fail or style violations occur, the orchestrator refines the prompt and regenerates. Once tests pass, the orchestrator updates the module's version and commits the code.

Integration

After each module is merged, downstream modules may need interface adjustments or retesting. The orchestrator tracks this via a module registry: whenever an interface version changes, it automatically queues updates to any modules that import it.

# Integration Registry Update
module: WalletConnector
version: 1.0.0
status: INTEGRATED
dependents:
  - TransactionService
  - UserDashboard
actions:
  - NOTIFY_DEPENDENTS
  - UPDATE_DOCS

Benefits

Maintaining Alignment Despite Context Limits

Each LLM prompt never needs the full app; it only needs the module's local code and the interface signatures of any collaborators. Because interfaces are stable and well-documented, the LLM won't propose incompatible changes.

Enabling Non-Technical Overseers

Product managers can define new modules and their user-facing goals without writing code. The TEAM tool translates these into AI prompts, and presents summarized outputs in human-friendly language for review.

Ensuring Code Quality and Consistency

Automated generation of docs and tests guarantees transparency and maintainability. The AI produces tests by analyzing the code and scenarios, which the TEAM workflow leverages to ensure reliability.

Facilitating Blockchain Application Development

By decomposing complex blockchain applications into manageable modules, TEAM makes it possible to build sophisticated Bitcoin inscription applications with AI assistance, despite the technical complexity involved.

Implementation

Prototype Tool Design

A prototype TEAM tool could be a Node.js CLI or web app ("TEAM-Dev") integrated with an LLM API. Its components would include:

Module Manifest

A JSON/YAML file (or GUI) where overseers list modules. Each entry has fields: name, description, inputs, outputs, dependencies, and human notes.

Orchestrator Engine

This reads the manifest and, for each module, gathers context: the module's spec, any existing code, and coding guidelines. It then constructs a prompt template and invokes the LLM.

Code Integration

The tool writes AI output to the code repo in designated folders. It updates the interface registry that other modules consult.

Automated Verification

A built-in pipeline runs tests. If tests fail or lint errors occur, the tool either flags the module for human review or automatically refines the prompt.

Human Dashboard

A simple web dashboard or CLI report summarizes progress and shows diffs of interface files so overseers see what changed at a glance.

Example: Bitcoin Inscription Application

Suppose the overseer wants a new "InscriptionCreator" module. They add a module card:

# Module Definition
name: InscriptionCreator
description: Creates ordinal inscriptions on Bitcoin
inputs:
  - content: string | Buffer
  - contentType: string
  - feeRate: number
outputs:
  - txid: string
  - inscriptionId: string
dependencies:
  - WalletConnector
  - TransactionBuilder

The TEAM tool auto-generates a prompt from this card and the definitions in the dependent modules' interfaces. The AI produces JavaScript code for InscriptionCreator plus docs/tests. The overseer reviews the generated interface summary and approves.
